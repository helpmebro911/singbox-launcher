# Спецификация: Rules Library — добавление правил из каталога шаблона в custom

## 1. Проблема

Сейчас на вкладке **Rules** два режима:

- **Встроенные (selectable)** — живут в `wizard_template.json`, состояние (вкл/выкл, outbound) в `selectable_rule_states`; порядок в UI и в `MergeRouteSection` жёстко завязан на порядок шаблона и двухфазный merge (сначала все selectable, потом все custom).
- **Пользовательские (custom)** — полные объекты в `custom_rules`, есть **↑ / ↓**, порядок сохраняется.

Пользователь не может произвольно чередовать «пресеты из шаблона» и свои правила в одной очереди. Отдельная доработка только порядка selectable не даёт единой модели «один список правил в маршруте».

---

## 2. Цель

1. Дать вход **«добавить из каталога»**: список пресетов из шаблона (`selectable_rules`) с **галочками**, по подтверждению — **копии в `custom_rules`** (rule / rules / rule_sets, outbound по умолчанию из шаблона и текущей логики).
2. **Сшивка с шаблоном по `label`:** в каталоге пресет идентифицируется **label** из шаблона. Если в шаблоне **переименовали** пресет (другой `label`) — это **новый** пункт каталога; старые записи в `custom_rules` **не трогаем**; пользователь при необходимости добавляет копию заново.
3. После перехода встроенные selectable **не участвуют в маршруте как отдельный слой** — это **источник пресетов**; **порядок и состав правил в `route`** задаёт **только** список **custom** (см. миграцию).
4. UX-вход: предпочтительно **кнопка на вкладке Rules** («Add from library» / согласованная строка), чтобы не плодить табы без необходимости; отдельная вкладка **Library** — допустима, если так проще навигации (зафиксировать в PLAN одним вариантом).

---

## 3. Требования

| ID | Требование |
|----|------------|
| R1 | UI на английском (CONSTITUTION); модальный список пресетов с мультивыбором (чекбоксы). |
| R2 | Список строится из **`TemplateData.SelectableRules`**. В строке: **label**; **description** — в tooltip или второй строкой, если помещается без перегруза. |
| R3 | По подтверждению: для каждого отмеченного пресета — **новая** запись в **`custom_rules`** (глубокая копия `rule` / `rules` / `rule_sets` и прочих полей, нужных для генерации конфига и диалога Edit). |
| R4 | **Дубликаты:** если в `custom_rules` уже есть правило с тем же **каноническим ключом каталога** — вторую копию **не** создавать. Ключ: **`library_source_label`** (см. R6) если задан; иначе fallback — сравнение **trimmed label** текущего правила с label пресета (чтобы не дублировать после миграции). Пользователю показать итог: например *«Added N, skipped M already present»* (строки в locale), без цепочки диалогов на каждое правило. |
| R5 | **Порядок вставки:** новые копии **в конец** списка `custom_rules` (сохраняется текущий порядок существующих правил). |
| R6 | Поле **`library_source_label`** (string, optional в JSON, **рекомендуется всегда выставлять при добавлении из каталога**): копия **label пресета на момент добавления**; в sing-box **не** уходит; при переименовании пресета в шаблоне значение в старых записях **не** обновляется. Позволяет отличить «добавлено из каталога» от ручного правила с совпадающим **label** после правки пользователем в редакторе. |
| R7 | **SRS:** копирование пресетов с `rule_set` — та же семантика, что у custom SRS (скачивание, локальные пути, кнопка в строке правила). |
| R8 | **Пустой список правил:** если после миграции/первого открытия на вкладке Rules нет ни одного custom — показать краткий **empty state** с призывом открыть каталог (одна primary-кнопка или ссылка на тот же диалог). |
| R9 | Документация: **docs/WIZARD_STATE.md**, при необходимости схема state, **docs/ARCHITECTURE.md** — поток Library → custom, merge, поле `library_source_label`. |

## 4. Миграция и обратная совместимость

| Вариант | Суть | Риск |
|--------|------|------|
| **A (рекомендуется)** | При загрузке state со «старым» форматом (есть значимые `selectable_rule_states` и политика «ещё не мигрировали»): для каждого пресета из шаблона создать **одну** запись в `custom_rules` (копия тела правила), перенести **enabled** и **selected_outbound** из сохранённого selectable; **порядок** мигрированных записей — порядок **в шаблоне**; **существующие** custom до миграции — **после** мигрированного блока (или наоборот — зафиксировать один раз в PLAN). Пометить state флагом/версией «library merge применён», очистить или игнорировать selectable в UI и в **MergeRouteSection**. | Нужно аккуратно описать порядок относительно уже существующих custom. |
| **B** | Переходный период: и каталог, и старый блок selectable на Rules. | Риск **двойного** правила в `route`, если включить и selectable, и копию в custom; в SPEC при варианте B обязательна политика: либо merge только из одного источника, либо жёсткое отключение selectable после первого добавления из каталога. |

- Старые `state.json` без `library_source_label` загружаются; поле опционально.
- **Вне scope:** автоматическое «обновить правило из шаблона» при изменении JSON шаблона (кроме сценария смены `label` = новый пресет).

---

## 5. Критерии приёмки

1. Из каталога можно добавить несколько пресетов; они в списке Rules как custom: вкл/выкл, outbound, **↑ / ↓**, Edit/Delete — как у остальных.
2. Повторное добавление того же пресета не создаёт дубликат; пользователь видит понятный итог (R4).
3. `route` строится из **custom** в порядке массива после выбранной миграции (без двойного учёта selectable).
4. Смена **label** в шаблоне не ломает загрузку state; старые копии на месте; в каталоге виден новый пресет.
5. Пустой список правил даёт понятный empty state (R8).
6. Обновлены docs (R9); сборка и тесты по **IMPLEMENTATION_PROMPT**.

---

## 6. Не делаем в этой задаче (явно)

- Поиск/фильтр по каталогу — опциональная **фаза 2**, если пресетов станет много.
- Drag-and-drop в каталоге — не требуется; порядок на вкладке Rules по-прежнему **↑ / ↓**.
- Синхронизация содержимого правила с обновлённым шаблоном по кнопке «Refresh from template» — отдельная фича, если понадобится.
