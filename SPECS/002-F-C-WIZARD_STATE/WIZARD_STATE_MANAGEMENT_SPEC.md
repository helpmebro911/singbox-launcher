# Спецификация: Хранение состояния визарда и режим управления (Wizard → Management)

**Кратко:**
Перевести текущий одноразовый визард в режим управления сохранёнными состояниями. Добавить промежуточный формат "wizard state" (JSON v1) и UI для просмотра/редактирования/восстановления состояний, чтобы можно было быстро изменять настройки и пересоздавать `config.json`.

## 🎯 Цели
- Сохранить текущее состояние визарда (включая выбранные опции шаблона, правила, URL источников) в отдельный файл(ы).
- Позволить пользователю управлять сохранёнными состояниями: список, загрузка, переименование, удаление, экспорт/импорт.
- При открытии визарда сначала проверять наличие сохранённого состояния и предлагать восстановить или создать новое.
- Сократить риск потери прогресса и упростить быстрые изменения/перегенерацию конфигурации.

---

## Текущее поведение

- При открытии визарда создаётся `WizardModel` в памяти (`ui/wizard/models/wizard_model.go`).
- При инициализации загружается `@ParserConfig` из `config.json` или из шаблона (`ui/wizard/business/loader.go`).
- Состояние визарда живёт только в оперативной памяти и теряется при закрытии окна.
- При сохранении генерируется финальный `config.json` из текущего состояния модели.

---

## Желаемое поведение

### Общий поток работы

Визард теперь работает с сохранёнными состояниями, что позволяет:
- Восстанавливать работу с того места, где остановились
- Создавать и переключаться между несколькими конфигурациями
- Сохранять прогресс без потери данных

### Описание сценариев

#### Сценарий 1: Первое открытие визарда
1. Пользователь открывает визард
2. `state.json` не существует
3. Визард загружает конфигурацию из `config.json` или шаблона (текущее поведение)
4. Пользователь работает с визардом, настраивает параметры
5. Пользователь нажимает "Save"
6. Состояние сохраняется в `state.json`

#### Сценарий 2: Повторное открытие визарда
1. Пользователь открывает визард
2. `state.json` существует
3. Показывается диалог выбора состояний (`state.json` выделен как "текущее")
4. Пользователь выбирает "Загрузить"
5. Состояние восстанавливается из `state.json`
6. Визард открывается с теми же настройками, что были при последнем сохранении

#### Сценарий 3: Работа с несколькими конфигурациями
1. Пользователь работает с визардом, настраивает конфигурацию "Работа"
2. Нажимает "Save As", вводит ID: `work-config`
3. Состояние сохраняется в `work-config.json`
4. При следующем открытии пользователь может выбрать нужную конфигурацию через "Read"
5.Пользователь изменяет настройки для конфигурации "Дом"
6. Нажимает "Save As", вводит ID: `home-config`
7. Состояние сохраняется в `home-config.json`

#### Сценарий 4: Сохранение с перезаписью текущего состояния
1. Пользователь работает с визардом, есть `state.json`
2. Пользователь изменяет настройки
3. Нажимает "Save"
4. Сразу показывается диалог "Save As" для сохранения предыдущего состояния
5. Пользователь вводит ID и комментарий (например, `backup-config`)
6. Предыдущее состояние сохраняется в `backup-config.json`
7. Текущее состояние автоматически сохраняется в `state.json`

#### Сценарий 5: Закрытие с несохранёнными изменениями
1. Пользователь работает с визардом, вносит изменения
2. Пользователь пытается закрыть визард
3. Показывается диалог подтверждения: "Сохранить изменения перед закрытием?"
4. Пользователь выбирает "Сохранить"
5. Состояние сохраняется в `state.json` (если файл существует, он перезаписывается без дополнительных диалогов)
6. Визард закрывается

#### Сценарий 6: Загрузка состояния с несохранёнными изменениями
1. Пользователь работает с визардом, вносит изменения (не сохраняет)
2. Пользователь нажимает "Read" для загрузки другого состояния
3. Показывается диалог выбора состояний
4. Пользователь выбирает состояние
5. Показывается диалог подтверждения: "Текущие изменения будут потеряны. Сохранить текущее состояние?"
6. Пользователь выбирает "Сохранить"
7. Показывается диалог "Save As", пользователь вводит ID и комментарий (например, `temp-config`)
8. Текущее состояние сохраняется в `temp-config.json`
9. Выбранное состояние загружается и копируется в `state.json`
10. Визард обновлён с загруженным состоянием, изменения не потеряны

## Архитектура решения

### 1. Структура данных состояния

Состояние визарда сериализуется из `WizardModel` в JSON-файл. Сохраняются все данные, необходимые для полного восстановления состояния:

- **ParserConfig** — JSON объект с конфигурацией парсера (из `model.ParserConfigJSON`, но сохраняется как объект, а не строка). Это единственный источник `@ParserConfig` для всего приложения (визард и кнопка "🔄 Update").
- **Примечание:** `SourceURLs` не сохраняется отдельно — URL'ы извлекаются из `parser_config.proxies` (поля `source` и `connections`) при восстановлении состояния.

- **ConfigParams** — параметры конфигурации (массив `ConfigParam`), используемые при генерации финального конфига. Каждый параметр имеет имя (путь в JSON в точечной нотации) и значение. Стандартные параметры: `route.final`, `experimental.clash_api.secret`.
- **SelectableRuleStates** — состояния правил из шаблона (из `model.SelectableRuleStates`), включая поле `type` (по умолчанию `"System"` для системных правил, но можно изменить на редактируемый тип)
- **CustomRules** — пользовательские правила (из `model.CustomRules`), включая поле `type` с типом правила

**Не сохраняются (генерируются/загружаются заново):**
- **TemplateData** — шаблон конфигурации загружается из файла (`wizard_template.json`) всегда при генерации конфига (определяет структуру и секции):
  - `Sections` (map[string]json.RawMessage) — секции конфига из шаблона (route, dns, inbounds, outbounds, experimental и т.д.). **Всегда используются из шаблона** для генерации конфига
  - `SectionOrder` ([]string) — порядок секций в шаблоне. **Всегда используется из шаблона** для сохранения порядка при генерации
  - `SelectableRules` ([]TemplateSelectableRule) — правила из шаблона (из `@SelectableRule` блоков). **В режиме редактирования НЕ используются** (правила берутся из `state.json`), **в режиме "Настроить заново" используются** для инициализации
  - `DefaultFinal` (string) — финальный outbound по умолчанию из секции `route` шаблона. **Используется как fallback**, если не задан в `config_params`
  - `HasParserOutboundsBlock` (bool) — есть ли маркер `@PARSER_OUTBOUNDS_BLOCK` в шаблоне. **Всегда используется из шаблона** для определения, куда вставлять сгенерированные outbounds
  - `OutboundsAfterMarker` (string) — элементы после маркера `@PARSER_OUTBOUNDS_BLOCK` (например, `direct-out`). **Всегда используется из шаблона** для добавления после сгенерированных outbounds
  - **В режиме редактирования:** шаблон загружается для генерации конфига (структура и секции), но правила берутся из `state.json`
  - **В режиме "Настроить заново":** шаблон используется для инициализации нового состояния (включая правила)
- **GeneratedOutbounds** — генерируются из ParserConfig при парсинге
- **TemplatePreviewText** — кэш preview, генерируется заново при необходимости

### 2. Формат хранения

**Расположение:** `<execDir>/bin/wizard_states/` (директория создаётся автоматически)

**Файлы:**
- `state.json` — текущее рабочее состояние (перезаписывается при сохранении)
- `<state_id>.json` — именованные сохранённые состояния (например, `my-config.json`, `work-vpn.json`)

**Правила именования:**
- ID состояния может содержать только буквы (a-z, A-Z), цифры (0-9), дефис (-) и подчёркивание (_)
- Максимальная длина ID: 50 символов
- Имена файлов формируются как `<state_id>.json` (без префиксов)

**Формат:** UTF-8 JSON с версионированием

```json
{
  "version": 1,
  "id": "my-config",
  "comment": "Конфигурация для работы",
  "created_at": "2026-01-30T12:00:00Z",
  "updated_at": "2026-01-30T15:30:00Z",
  
  "parser_config": {
      "version": 4,
      "proxies": [
        {
          "source": "https://example.com/sub.txt"
        }
      ],
      "outbounds": [],
      "parser": {
        "reload": "1h",
        "last_updated": "2026-01-30T12:00:00Z"
      }
    }
  },
  "config_params": [
    {
      "name": "route.final",
      "value": "proxy-out"
    },
    {
      "name": "experimental.clash_api.secret",
      "value": "my-secret-token-12345"
    }
  ],
  "selectable_rule_states": [
    {
      "type": "System",
        "rule": {
        "label": "Block ads",
        "description": "Block advertisement domains",
        "raw": {
          "domain_suffix": ["ads.example.com"],
          "action": "reject"
        },
        "default_outbound": "reject",
        "has_outbound": false,
        "is_default": true
      },
      "enabled": true,
      "selected_outbound": "reject"
    }
  ],
  "custom_rules": [
    {
      "type": "Domains/URLs",
      "rule": {
        "label": "Custom rule",
        "description": "Route example.com to proxy",
        "raw": {
          "domain": ["example.com"],
          "outbound": "proxy-out"
        },
        "default_outbound": "proxy-out",
        "has_outbound": true,
        "is_default": false
      },
      "enabled": true,
      "selected_outbound": "proxy-out"
    }
  ]
} 
```

**Примечания:**
- Поле `version` всегда равно 1 в первой версии (миграции не поддерживаются)
- Поля `created_at` и `updated_at` в формате RFC3339 для отображения в UI
- Поле `id` опционально для `state.json` (имя файла является идентификатором), обязательно для именованных состояний и должно совпадать с именем файла
- Поле `comment` опционально — комментарий/описание состояния для пользователя
- `parser_config` хранится как JSON объект (не строка)
- `parser_config` является единственным источником `@ParserConfig` для всего приложения:
  - Используется визардом при работе с конфигурацией
  - Используется при нажатии кнопки "🔄 Update" в главном окне для обновления конфигурации из подписок и для регулярных обновлений
  - При сохранении финальной конфигурации этот блок включается в `config.json` как комментарий `/** @ParserConfig */`
- **Важно:** `@ParserConfig` больше не хранится в `config.json` (только в `state.json`). При загрузке состояния визард и функция обновления используют этот блок вместо `config.json`
- `config_params` содержит параметры конфигурации (например, `route.final`, `experimental.clash_api.secret`)
- `selectable_rule_states` и `custom_rules` содержат полную структуру `RuleState` с `TemplateSelectableRule`, включая поле `type`

### 3. Go-структуры

```go
// WizardStateFile представляет сериализуемое состояние визарда
type WizardStateFile struct {
    Version                  int                    `json:"version"`
    ID                       string                 `json:"id,omitempty"` // Опционально для state.json, обязательно для именованных состояний
    Comment                  string                 `json:"comment,omitempty"`
    CreatedAt                time.Time              `json:"created_at"`
    UpdatedAt                time.Time              `json:"updated_at"`
    
    ParserConfig             config.ParserConfig    `json:"parser_config"`
    ConfigParams             []ConfigParam          `json:"config_params"`
    SelectableRuleStates     []PersistedRuleState   `json:"selectable_rule_states"`
    CustomRules              []PersistedRuleState   `json:"custom_rules"`
}

// ConfigParam представляет параметр конфигурации
type ConfigParam struct {
    Name  string `json:"name"`  // Путь к параметру в точечной нотации (например, "route.final", "experimental.clash_api.secret")
    Value string `json:"value"`  // Значение параметра (может быть пустой строкой для использования значения по умолчанию)
}

// PersistedRuleState - сериализуемая версия RuleState
type PersistedRuleState struct {
    Type             string                          `json:"type"` // "System" по умолчанию для предустановленных (можно изменить на редактируемый тип), или "IP Addresses (CIDR)", "Domains/URLs", "Processes", "Custom JSON" для пользовательских
    Rule             PersistedTemplateSelectableRule `json:"rule"`
    Enabled          bool                            `json:"enabled"`
    SelectedOutbound string                          `json:"selected_outbound"`
}

// PersistedTemplateSelectableRule - сериализуемая версия TemplateSelectableRule
type PersistedTemplateSelectableRule struct {
    Label           string                 `json:"label"`
    Description     string                 `json:"description"`
    Raw             map[string]interface{} `json:"raw"`
    DefaultOutbound string                 `json:"default_outbound"`
    HasOutbound     bool                   `json:"has_outbound"`
    IsDefault       bool                   `json:"is_default"`
}

// WizardStateMetadata - метаданные состояния для списка
type WizardStateMetadata struct {
    ID        string    `json:"id"`
    Comment   string    `json:"comment,omitempty"` // Комментарий из файла состояния (может быть пустым)
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
    IsCurrent bool      `json:"is_current"` // true если это state.json
}
```

## Функциональные требования

### 1. Сохранение состояния

#### Кнопка "Save"
- Сохраняет текущее состояние в `state.json`
- **Обновление `updated_at`:** устанавливается в текущее время (time.Now()) при сохранении
- Если `state.json` уже существует:
  - Сразу показать диалог "Save As" для сохранения предыдущего состояния
  - После сохранения предыдущего состояния автоматически сохранить текущее в `state.json`
  - При отмене диалога "Save As" — операция отменена, `state.json` не изменяется
- Если `state.json` не существует — сохранить без диалогов

#### Кнопка "Save As"
- Всегда запрашивает ID и комментарий для нового состояния
- Валидирует ID (только буквы, цифры, дефис, подчёркивание; макс. 50 символов)
- Если ID уже существует — показывается предупреждение в диалоге: "Состояние с таким ID уже существует. Будет перезаписано." Кнопка "Сохранить" перезаписывает существующее состояние
- Сохраняет состояние в файл `<state_id>.json`
- **Обновление `created_at` и `updated_at`:** устанавливаются в текущее время (time.Now()) при создании нового состояния
- Не перезаписывает `state.json` (кроме случая, когда вызывается из "Save" с существующим `state.json`)

**Важно:** `updated_at` обновляется **только при явном сохранении состояния** через кнопки "Save" или "Save As". При автоматических обновлениях через кнопку "🔄 Update" в главном окне обновляется только `parser_config.parser.last_updated`, но **не** `updated_at` в `state.json` (так как состояние не сохраняется автоматически).

### 2. Загрузка состояния

#### При открытии визарда

**Режим редактирования (по умолчанию):**
- Если существует `state.json` — загружается сохраненное состояние
- Все данные (правила, параметры, ParserConfig) берутся из `state.json`
- Шаблон загружается только для генерации финального конфига (структура и секции)

**Режим "Настроить заново":**
- Опция в UI, которая игнорирует `state.json`
- Используется только шаблон для инициализации нового состояния
- Полезно, когда шаблон обновился и нужно начать с чистого листа

**Инициализация `config_params` в режиме "Настроить заново":**
- Инициализируется так же, как при создании конфига из визарда:
  - `route.final`: значение из `TemplateData.DefaultFinal` (если есть в шаблоне), иначе пустая строка `""`
  - `experimental.clash_api.secret`: **не инициализируется** — будет сгенерирован автоматически при первом сохранении конфига через `SaveConfigWithBackup` (как сейчас при создании конфига из визарда)
- После инициализации пользователь может изменить значения через UI визарда

**Инициализация `config_params` в режиме "Настроить заново":**
- Инициализируется так же, как при создании конфига из визарда:
  - `route.final`: значение из `TemplateData.DefaultFinal` (если есть в шаблоне), иначе пустая строка `""`
  - `experimental.clash_api.secret`: **не инициализируется** — будет сгенерирован автоматически при первом сохранении конфига через `SaveConfigWithBackup` (как сейчас при создании конфига из визарда)
- После инициализации пользователь может изменить значения через UI визарда

#### Детали реализации при открытии визарда
1. Проверить наличие файла `state.json` в `bin/wizard_states/` и других сохранённых состояний
2. Если есть `state.json` или другие сохранённые состояния:
   - Показать диалог выбора состояний (`load_state_dialog.go`)
   - `state.json` автоматически выделен как "текущее" (если существует)
   - Варианты: "Загрузить" (выбранное состояние или `state.json`), "Настроить заново", "Отмена"
   - При выборе "Загрузить" — загрузить выбранное состояние (или `state.json`) и восстановить `WizardModel`
   - При выборе "Настроить заново" — игнорировать все сохранённые состояния, использовать только шаблон для инициализации нового состояния
   - При выборе "Отмена" — закрыть визард
3. Если сохранённых состояний нет:
   - Использовать текущую логику (загрузка из `config.json`/шаблона)

#### Кнопка "Read" (в начале визарда)
- Открывает диалог выбора состояния из списка всех сохранённых состояний
- При выборе состояния:
  - Проверить наличие несохранённых изменений (`hasUnsavedChanges()`)
  - Если есть несохранённые изменения:
    - Показать диалог подтверждения: "Текущие изменения будут потеряны. Сохранить текущее состояние?"
    - Варианты: "Сохранить" (открыть диалог "Save As"), "Не сохранять", "Отмена"
    - При выборе "Сохранить" — открыть диалог "Save As" для сохранения текущего состояния, затем продолжить загрузку
    - При выборе "Не сохранять" — продолжить загрузку без сохранения (потерять изменения)
    - При выборе "Отмена" — отменить операцию, остаться в визарде
  - Если изменений нет или пользователь подтвердил загрузку:
    - Загрузить выбранное состояние в память
    - Скопировать в `state.json` как "текущее"
  - Восстановить `WizardModel` из загруженного состояния (см. детальную последовательность ниже)
  - Установить `hasChanges = false`

#### Детальная последовательность восстановления WizardModel

При восстановлении состояния из `WizardStateFile` выполняется следующая последовательность:

1. **Загрузить шаблон** (`TemplateData`) из файла (`wizard_template.json`)
   - Шаблон используется для генерации конфига (структура и секции)

2. **Восстановить `parser_config`:**
   - `model.ParserConfig = stateFile.ParserConfig`
   - Сериализовать `parser_config` в JSON строку: `model.ParserConfigJSON = SerializeParserConfig(stateFile.ParserConfig)`

3. **Извлечь `SourceURLs` из `parser_config`:**
   - Пройти по `parser_config.proxies`

**✅ Выполнено:** Структура упрощена, используется `parser_config.proxies` напрямую.
   - Собрать все значения `source` и `connections` в многострочный текст
   - `model.SourceURLs = объединенные URL'ы через "\n"`

4. **Восстановить `config_params` и маппинг в модель:**
   - Найти параметр с `name = "route.final"` в `config_params`
   - Если найден: `model.SelectedFinalOutbound = param.Value`
   - Если не найден или `value = ""`: `model.SelectedFinalOutbound = TemplateData.DefaultFinal` (fallback из шаблона)
   - Остальные параметры из `config_params` сохраняются для применения при генерации конфига (например, `experimental.clash_api.secret`)

5. **Инициализировать `TemplateSectionSelections`:**
   - Для каждой секции из `TemplateData.SectionOrder`: `model.TemplateSectionSelections[key] = true`
   - Пользователь не может изменять выбор секций в UI, поэтому всегда все секции включены

6. **Восстановить `SelectableRuleStates`:**
   - Загрузить полностью из `stateFile.SelectableRuleStates`
   - Восстановить каждое правило с полями: `rule.raw`, `rule.label`, `rule.description`, `type`, `enabled`, `selected_outbound`

7. **Восстановить `CustomRules`:**
   - Загрузить полностью из `stateFile.CustomRules`
   - Восстановить каждое правило с полями: `rule.raw`, `rule.label`, `rule.description`, `type`, `enabled`, `selected_outbound`

8. **Запустить парсинг:**
   - Вызвать функцию парсинга для генерации `model.GeneratedOutbounds` из `model.ParserConfig`

9. **Синхронизировать GUI:**
   - Обновить все виджеты в соответствии с восстановленной моделью

### 3. Закрытие визарда

- При закрытии визарда (кнопка "Close" или крестик):
  - Если были изменения в состоянии (отслеживать флаг `hasChanges`):
    - Показать диалог подтверждения: "Сохранить изменения перед закрытием?"
    - Варианты: "Сохранить", "Не сохранять", "Отмена"
    - При выборе "Сохранить":
      - Сохранить текущее состояние в `state.json` (перезаписать, если файл существует, без дополнительных диалогов)
      - Закрыть визард
    - При выборе "Не сохранять" — закрыть без сохранения (потерять изменения)
    - При выборе "Отмена" — остаться в визарде
  - Если изменений нет (`hasChanges = false`) — закрыть без диалога

### 5. Валидация и обработка ошибок

#### Валидация ID состояния
- Разрешены только: буквы (a-z, A-Z), цифры (0-9), дефис (-), подчёркивание (_)
- Максимальная длина: 50 символов
- При невалидном ID — показать ошибку с описанием правил

#### При загрузке файла состояния

**Валидация перед загрузкой:**
- Проверка версии формата (только версия 1, при других версиях — ошибка с сообщением "Неподдерживаемая версия формата")
- Валидация JSON-структуры (парсинг должен пройти успешно)
- Проверка обязательных полей: `version`, `parser_config`, `config_params`, `selectable_rule_states`, `custom_rules`
- Валидация `id` (если присутствует): только разрешённые символы, макс. 50 символов, совпадение с именем файла

**✅ Выполнено:** Валидация обновлена для проверки полей `version`, `proxies`, `outbounds`, `parser` напрямую в `parser_config`.

**Обработка ошибок при восстановлении:**

1. **Невалидный `parser_config`:**
   - Если `parser_config` не является валидным JSON объектом или не содержит обязательные поля (`version`, `proxies`, `outbounds`, `parser`)
   - Действие: показать ошибку "Невалидный parser_config в файле состояния", fallback на загрузку из `config.json`/шаблона



2. **Невалидный параметр в `config_params`:**
   - Если параметр имеет невалидное имя (например, `"invalid.path"` или пустое имя)
   - Действие: пропустить невалидный параметр, показать предупреждение в логах, продолжить загрузку

3. **Отсутствующий параметр `route.final` в `config_params`:**
   - Если параметр `route.final` отсутствует
   - Действие: использовать значение по умолчанию из шаблона (`TemplateData.DefaultFinal`), продолжить загрузку

4. **Невалидное правило в `selectable_rule_states` или `custom_rules`:**
   - Если правило не содержит обязательных полей (`rule`, `enabled`, `selected_outbound`) или `rule.raw` не является валидным JSON объектом
   - Действие: пропустить невалидное правило, показать предупреждение в логах, продолжить загрузку остальных правил

5. **Правило с `type="System"` не соответствует шаблону:**
   - Если правило имеет `type="System"`, но `rule.raw` не соответствует ни одному правилу из шаблона
   - Действие: оставить правило как есть (возможно, пользователь изменил тип на редактируемый), продолжить загрузку

6. **Ошибка загрузки шаблона:**
   - Если шаблон не может быть загружен из файла
   - Действие: показать ошибку "Не удалось загрузить шаблон", fallback на загрузку из `config.json`/шаблона

7. **Ошибка парсинга после восстановления:**
   - Если после восстановления состояния парсинг `parser_config` завершается с ошибкой
   - Действие: показать ошибку "Ошибка парсинга после восстановления состояния", но оставить состояние загруженным (пользователь может исправить)

**Общая стратегия обработки ошибок:**
- Критические ошибки (невалидный JSON, неподдерживаемая версия) — показать ошибку и fallback на загрузку из `config.json`/шаблона
- Некритические ошибки (невалидные параметры, правила) — пропустить проблемные элементы, показать предупреждение в логах, продолжить загрузку
- При любой ошибке загрузки — показать понятное сообщение пользователю с описанием проблемы

#### При сохранении
- Проверка размера файла (максимум 256 KB, предупреждение при превышении)
- Обработка ошибок записи (недостаточно прав, диск заполнен и т.д.)
- При "Save As" с существующим ID:
  - Показать предупреждение в диалоге "Save As": "Состояние с таким ID уже существует. Будет перезаписано."
  - Кнопка "Сохранить" перезаписывает существующее состояние без дополнительных диалогов

## Диалоговые окна

Все диалоговые окна сведены к минимуму для упрощения UX. Всего используется **3 типа диалогов**:

### 1. Диалог выбора состояния (`load_state_dialog.go`)

**Когда появляется:**
- При открытии визарда, если существует `state.json` или есть другие сохранённые состояния
- При нажатии кнопки "Read"

**Содержимое:**
- Простой список имён файлов состояний:
  - Отображаются только имена файлов (например, `state.json`, `my-config.json`)
  - Для `state.json` добавляется метка "(Current)"
  - Высота списка: ~5 строк с прокруткой
  - **Оптимизация:** используется `ListWizardStateNames()` - читаются только имена файлов из директории, без чтения содержимого файлов
- Кнопки: "Load", "New", "Cancel"
  - "Load" — загрузить выбранное состояние (или `state.json`, если ничего не выбрано)
  - "New" — игнорировать все сохранённые состояния, начать с шаблона
  - "Cancel" — закрыть диалог без загрузки

**Особенности:**
- При открытии визарда: если есть `state.json`, он автоматически выделен как "текущее"
- При "Read": если есть несохранённые изменения, сначала показывается диалог подтверждения (см. ниже)
- Для отображения списка читаются только имена файлов из директории `bin/wizard_states/`, содержимое файлов не читается до момента загрузки состояния

### 2. Диалог сохранения состояния (`save_state_dialog.go`)

**Когда появляется:**
- При нажатии кнопки "Save As"
- При нажатии кнопки "Save", если `state.json` уже существует (для сохранения предыдущего состояния)

**Содержимое:**
- Поле ввода ID (обязательное):
  - Валидация: только буквы (a-z, A-Z), цифры (0-9), дефис (-), подчёркивание (_)
  - Максимальная длина: 50 символов
  - Если ID уже существует — показывается предупреждение в диалоге: "Состояние с таким ID уже существует. Будет перезаписано."
  - Кнопка "Сохранить" остается активной и перезаписывает существующее состояние
- Поле ввода комментария (необязательное):
  - Текстовая область для описания состояния
- Кнопки: "Сохранить", "Отмена"

**Особенности:**
- При "Save" с существующим `state.json`: после сохранения предыдущего состояния автоматически сохраняется текущее в `state.json`
- При "Save As": сохраняется только новое именованное состояние, `state.json` не изменяется

### 3. Диалог подтверждения (универсальный)

**Когда появляется:**
- При нажатии "Read" с несохранёнными изменениями
- При закрытии визарда с несохранёнными изменениями

**Содержимое:**
- Текст сообщения (зависит от контекста):
  - При "Read": "Текущие изменения будут потеряны. Сохранить текущее состояние?"
  - При закрытии: "Сохранить изменения перед закрытием?"
- Кнопки: "Сохранить", "Не сохранять", "Отмена"
  - "Сохранить" — открыть диалог "Save As" (для "Read") или сохранить в `state.json` (для закрытия)
  - "Не сохранять" — продолжить операцию без сохранения (потерять изменения)
  - "Отмена" — отменить операцию, остаться в визарде

**Особенности:**
- При "Read" → "Сохранить": открывается диалог "Save As", после сохранения продолжается загрузка выбранного состояния
- При закрытии → "Сохранить": сохраняется в `state.json` (если файл существует, он перезаписывается без дополнительных диалогов)

## Нефункциональные требования

- **Производительность:** операции Load/Save должны выполняться быстро (< 100ms для типичных файлов)
- **Безопасность:** файлы состояний хранятся локально, без передачи в облако
- **Читаемость:** JSON файлы форматируются с отступами для удобства отладки
- **Обратная совместимость:** если сохранённых состояний нет — поведение без изменений (используется текущая логика)
- **Версионирование:** в первой версии поддерживается только версия 1, миграции не реализуются
