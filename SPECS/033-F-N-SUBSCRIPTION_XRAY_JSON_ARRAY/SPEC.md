# SPEC: Подписки — JSON-массив полных конфигов Xray/V2Ray (включая dialerProxy / jump на SOCKS)

## 1. Проблема

### 1.1 Контекст

Часть провайдеров отдаёт по URL подписки **не** base64/plain со списком URI, а **JSON-массив**, каждый элемент которого — **полный конфиг в стиле Xray/V2Ray**: `log`, `dns`, `inbounds`, `outbounds`, `routing`, `remarks` и т.д.

Репрезентативный кейс (пользовательский образец):

- корень: `[ { ... }, { ... }, ... ]`;
- основной исходящий прокси: `protocol: vless`, `settings.vnext[]`, `streamSettings` с `security: reality`, `flow` в пользователе (`xtls-rprx-vision`);
- **цепочка**: в `streamSettings.sockopt` указан **`dialerProxy`** (ссылка на тег другого outbound, например `ru-upstream`);
- **jump**: отдельный outbound `protocol: socks` с уникальными **на каждый элемент массива** `address`/`port`/учётными данными (не один общий jump на всю подписку);
- человекочитаемое имя профиля: поле **`remarks`** на уровне элемента конфига.

Текущий парсер:

- в `DecodeSubscriptionContent` ответ, начинающийся с `[`, отклоняется как «не подписка»;
- пайплайн нод заточен под **строки URI** → sing-box-shaped `ParsedNode` / `buildOutbound`;
- `GenerateNodeJSON` **не** эмитит поле **`detour`** (Dial Fields sing-box), то есть цепочка «jump → основной» в сгенерированный `config.json` не попадает;
- модель **одна нода ↔ один outbound** не отражает пару SOCKS + VLESS с пер-серверным jump.

Связанная задача **`016-F-C-SUBSCRIPTION_JSON_ARRAY`** описывает JSON-массив конфигов в основном как **sing-box** (`type` в outbounds); **закрыта без реализации**, SPEC остаётся референсом. Настоящая задача покрывает **другой диалект** того же «массива конфигов»: **Xray JSON**, включая **`dialerProxy` → эквивалент `detour` в sing-box**.

### 1.2 Цель

Поддержать загрузку подписок в формате **JSON-массива полных Xray-подобных конфигов**: из каждого элемента массива извлекать **ровно одну пользовательскую ноду** (основной прокси для селектора), при необходимости восстанавливая **цепочку через SOCKS** так, как ожидает sing-box (**отдельный outbound jump + `detour` на основном**).

Поведение для пользователя должно быть сопоставимо с обычной подпиской: те же лимиты, префиксы/постфиксы тегов, фильтры Skip, `MakeTagUnique`, визард CheckURL / ApplyURL.

## 2. Требования

### 2.1 Распознавание формата

- После `FetchSubscription` и прохода через `DecodeSubscriptionContent` (или эквивалентный путь в визарде) тело, начинающееся с **`[`** и являющееся валидным JSON-массивом, **не** считается ошибкой «JSON instead of subscription».
- Дальнейшая обработка: ветка **«массив конфигов»** в загрузчике нод (`LoadNodesFromSource` и парсер визарда), **без** построчного split по URI.
- Если массив распознан, но ни один элемент не дал нод — это **0 нод** (как пустая подписка), не обязательно ошибка; при полностью невалидной структуре JSON — одна понятная ошибка.

**MVP задачи 033:** поддерживается только разбор элементов массива в диалекте **Xray** (см. **§2.2**). Массив конфигов в формате **sing-box** (`type` в outbounds), описанный в **`016-F-C-SUBSCRIPTION_JSON_ARRAY`**, в MVP **не** реализуется — отдельный подэтап или follow-up после 033.

### 2.2 Определение «Xray-элемента»

Элемент массива считать кандидатом в формате **Xray full config**, если выполняется минимум одно из (эвристика, документировать в коде):

- в `outbounds[]` у объектов есть поле **`protocol`** (а не только sing-box **`type`**), или
- присутствуют **`settings.vnext`** / **`settings.servers`** в стиле V2Ray для vless/vmess/trojan и т.п.

В MVP элементы, которые **не** распознаны как Xray (например, только sing-box `type` без Xray-`protocol`), **пропускаются** с записью в `debuglog`; разбор sing-box-массива — в follow-up по **016**.

### 2.3 Извлечение основной ноды

- Из каждого элемента массива выбрать **один** «основной» прокси-аутбаунд: тот, через который пользователь должен ходить в интернет (в эталонном файле — тег вроде `proxy`, **VLESS + REALITY**).
- Правило выбора задать явно в **PLAN** (например: аутбаунд с `dialerProxy`/цепочкой; при отсутствии — единственный `vless`/`vmess`/…; при неоднозначности — приоритет по тегу или порядку + лог `debuglog`).

Служебные типы Xray (`freedom`, `blackhole`, `dns` и аналоги) в список пользовательских нод **не** включать.

### 2.4 Jump (dialerProxy) и пер-серверная уникальность

- Если у основного outbound в `streamSettings.sockopt` (или эквивалентном месте в нормализованной структуре) указан **`dialerProxy`** (строка-тег):
  - найти в том же элементе массива outbound с этим `tag` и типом **socks** (на первом этапе — только SOCKS5, как в эталоне);
  - сформировать **два** sing-box outbound в результирующем конфиге: **jump** (без `detour`) и **основной** с **`detour`**, равным тегу jump-outbound.
- Если у основного outbound **указан** `dialerProxy`, но outbound с этим `tag` **не найден**, или его тип не **socks**: **элемент не даёт ноды** (пропуск), с **`WarnLog`**; остальные элементы обрабатываются как обычно. При **отсутствии** `dialerProxy` у основного outbound цепочка не строится, нода — один outbound (как в PLAN §3).
- У jump (SOCKS) **допускается отсутствие логина/пароля** (пустой или отсутствующий `users` в Xray): sing-box outbound без `username`/`password`.
- Учётные данные и адрес jump **никогда не смешивать** между элементами массива: каждая пара берётся **только** из своего объекта конфига.

Имена тегов:

- после разбора должны проходить через существующие механизмы **TagPrefix/TagPostfix/TagMask** и **`MakeTagUnique`**;
- гарантировать, что тег jump и тег основного outbound **стабильно уникальны** в рамках подписки и не конфликтуют с другими источниками (стратегия: суффиксы, префикс из `remarks`, см. **PLAN**).

### 2.5 Конвертация Xray outbound → поля sing-box

- Преобразовать **VLESS** (и при необходимости другие типы в рамках того же эталона) из Xray JSON в ту же семантику, что даёт текущий путь URI → `buildOutbound` / `GenerateNodeJSON`: `server`, `server_port`, `uuid`, `flow`, `tls` с **reality** (`public_key`, `short_id`, `server_name`, uTLS fingerprint), `transport` при нетривиальном транспорте.
- Поля с другими именами Xray (`publicKey` → `public_key`, `serverName` → `server_name`, `shortId` → `short_id`, `fingerprint` → `utls.fingerprint`, и т.д.) маппить согласно документации sing-box и существующим хелперам в `node_parser_transport.go` где возможно **переиспользовать** логику, а не дублировать.

### 2.6 Генерация JSON (`GenerateNodeJSON` и вызывающий код)

- Для ноды с цепочкой: либо **расширить** `GenerateNodeJSON` / генератор, либо ввести соседний API, который выдаёт **одну или две** JSON-строки для вставки в массив `outbounds` лаунчера:
  - первая строка — jump (`type: socks`, …);
  - вторая — основной протокол с полем **`detour`** ([Dial Fields](https://sing-box.sagernet.org/configuration/shared/dial/)).
- Порядок вставки: jump **до** основного, чтобы тег `detour` резолвился.
- Селекторы и фильтры по-прежнему должны ссылаться на тег **основной** ноды (тот, что видит пользователь как «один сервер»); jump **не** должен появляться как отдельный выбираемый peer в UI без отдельного решения (по умолчанию — только тег основного в списке нод селектора).

### 2.7 Метаданные ноды

- **`Label`**: приоритет полного текста **`remarks`** элемента конфига; при отсутствии — тег основного Xray-outbound или сгенерированный идентификатор (`xray-{индекс}`). Используется в комментарии `//` к сгенерированному outbound и в полях ноды для фильтров/отображения там, где смотрят на `Label`.
- **Теги sing-box** (поле `tag` у outbound): при непустом **`remarks`** — slug из `remarks` как тег основного outbound; при цепочке — второй outbound с суффиксом **`_jump_server`**; при пустом `remarks` — **`xray-{индекс}`** и **`xray-{индекс}_jump_server`**. Далее — **`tag_prefix` / `tag_postfix` / `tag_mask`**, **`MakeTagUnique`** (см. **PLAN §7** и реализацию).
- Поля `ParsedNode` для цепочки: **`Jump`** (`ParsedJump`) — по **PLAN**; публичный API других пакетов не ломать без необходимости.

### 2.8 Ограничения и последующие итерации

- **Share URI** (`ShareURIFromOutbound` и пункт меню «Копировать ссылку»): для ноды с jump **не обязано** выдавать одну воспроизводимую ссылку в рамках первой итерации; допустимо явно вернуть «не поддерживается» или кодировать только основной hop без jump — зафиксировать в **PLAN/TASKS** выбранное поведение.
- Одиночный JSON-объект `{...}` как тело подписки — **вне scope** (как в 016), если не решено иначе отдельно.
- Глубокая поддержка всех протоколов Xray и всех вариантов `routing` из файла **не** требуется: только извлечение прокси-нод для лаунчера.
- Jump по `dialerProxy`: в **033** заложена цепочка **SOCKS** → основной outbound; расширение на **VLESS** и др. — **`SPECS/036-F-C-XRAY_JUMP_ANY_PROTOCOL/SPEC.md`** (VLESS как hop реализован в **036**).

### 2.9 Критерии приёмки

1. Подписка URL, отдающая JSON-массив в стиле эталонного Xray-файла, успешно загружается; ошибка «subscription URL returned JSON configuration instead of subscription list» для валидного массива **не** возникает.
2. Из каждого элемента массива получается **одна** логическая нода в списке (с корректным именем из `remarks` где есть).
3. При наличии `dialerProxy` + SOCKS в том же конфиге в сгенерированном `config.json` присутствуют **два** outbound с корректным **`detour`** на jump; sing-box принимает конфиг (проверка: ручная или интеграционный тест на структуру JSON).
4. Разные элементы массива с разными jump не перепутывают адреса/учётки.
5. Регрессии: base64 и plain URI подписки. JSON-массив **sing-box** (016) в MVP не входит — отдельный follow-up.
6. Логи `debuglog`, сообщения пользователю — по **CONSTITUTION** (английский для UI).
